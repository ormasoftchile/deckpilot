## Proposed Verification Tiers

### Tier 1 — String Assertions (Available Now)

Use `parseSlides()` and assert on `slide.html` string content. Zero new infrastructure. Catches plugin logic errors and markdown-it regressions.

**Cost:** None | **Value:** High

### Tier 2 — DOM Structural Assertions (Recommended)

Add `node-html-parser` dev dependency. Write helper `test/unit/helpers/renderSlide.ts` to parse final HTML. Unlocks nesting, sibling, and depth assertions.

**Cost:** One dev dependency + one helper file | **Value:** Very High

**Test example:**
```typescript
it(':::columns wraps :::left and :::right as children', () => {
  const root = renderSlideToHtml(':::columns\n:::left\nA\n:::\n:::right\nB\n:::\n:::');
  const columns = root.querySelector('.layout-columns');
  expect(columns!.querySelector('.layout-left')).to.exist;
  expect(columns!.querySelector('.layout-right')).to.exist;
});
```

### Tier 3 — Visual Regression / Screenshot Diffs (Rejected)

Playwright or Puppeteer visual testing. **Not recommended.**

**Reasons:**
- Webview rendering requires VS Code CSS variables — headless rendering produces incorrect output
- Playwright VS Code testing is slow, flaky, high maintenance
- Visual regression is lower priority than structural correctness for layout changes
- Revisit only if project adds standalone HTML renderer

## Practical First Step

**Write `test/unit/parser/slideRenderingPipeline.test.ts`** with:
1. `:::center` renders wrapper + inner markdown
2. `:::columns` renders `.layout-columns > .layout-left` and `.layout-columns > .layout-right`
3. `:::advanced` renders `<details class="disclosure-advanced">` with `<summary>`
4. `:::columns` with fragments assigns correct indices
5. Slide with no directives renders unchanged (regression guard)

**This is the MVP test suite for layout changes.** Uses existing test runner (`npm run test:unit`), no new infrastructure beyond Tier 1.
