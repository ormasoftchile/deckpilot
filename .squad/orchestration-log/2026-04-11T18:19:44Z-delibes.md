## Objective

Create comprehensive end-to-end test coverage for the layout rendering pipeline (`transformLayoutDirectives()` → `markdown-it` → `processFragments()`) to prevent regressions when modifying layout classes, HTML structure, or rendering semantics.

## Deliverables

### 1. Test File: `test/unit/parser/slideRenderingPipeline.test.ts`

**Scope:** 38 tests across 6 describe blocks

- **`:::center` directive** (7 tests) — wrapper class `.layout-center`, content preservation, heading inside, div tag balance
- **`:::columns` directive** (8 tests) — outer wrapper, left/right columns, content preservation, div balance, left-before-right order
- **`:::advanced` disclosure** (5 tests) — `<details>` element, class `.disclosure-advanced`, `<summary>Advanced</summary>`, content, close tag
- **`:::optional` disclosure** (5 tests) — wrapper class `.step-optional`, badge span with "Optional" label, content, close tag
- **Plain slides (no layout)** (3 tests) — heading, paragraph, bullet list, inline code; verify no wrapper injected
- **Multiple slides** (7 tests) — 1/2/3 slide splitting, independent rendering, sequential indices, mixed layouts per slide
- **Edge cases** (3 tests) — empty string, directive-only (no body), whitespace-only

**All tests use Tier 1 string assertions** (`contain()` / `regex`) on `parseSlides()` output. Zero new runtime dependencies.

### 2. Mock Infrastructure: `test/unit/helpers/vscode-mock.cjs`

**Problem:** `parseSlides()` transitively imports the renderer barrel (`src/renderers/index.ts`) which has `import * as vscode from 'vscode'`. Vscode is unavailable in headless Mocha.

**Solution:** CJS module that patches `Module._resolveFilename` and `require.cache` to intercept `vscode` requires and provide a minimal stub with:
- `workspace.workspaceFolders` (returns `[]`)
- `workspace.fs.readFile()` (no-op stub)
- `Uri.file()` (returns path as-is)
- `Uri.parse()` (returns URI as-is)

**Intentionally minimal:** This is NOT a general-purpose vscode mock. Do not extend it to enable tests that exercise vscode behavior. If vscode behavior needs testing, use `@vscode/test-electron`.

**Integration:** `package.json` `test:unit` script includes `--require ./test/unit/helpers/vscode-mock.cjs --require ts-node/register` before mocha specs.

## Technical Decisions

### String Assertions (Tier 1 Only)

Per team decision in `decisions.md` (Test Verification Strategy, Delibes 2026-04-11):
- **Tier 1:** String `contain()` / regex assertions on `parseSlides()` output ✅ (this file)
- **Tier 2:** `node-html-parser` for DOM structural checks (not yet approved)

Rationale for Tier 1:
- Zero new runtime dependencies
- Sufficient to anchor class names and tag presence
- Nesting/sibling order checked by relative index positions (e.g., `layout-left` appears before `layout-right` in HTML string)

Tier 2 adoption criterion:
- A layout produces deeply nested structures where sibling order matters
- A test needs "element A is child of element B" semantics (not just both present)

### Going Forward

**Rule:** Any new layout directive added to `layoutDirectivePlugin.ts` MUST have a corresponding entry in `slideRenderingPipeline.test.ts` asserting the final HTML output structure.

This is the safety net for all layout improvements (column ratios, label overrides, onboarding centering).
